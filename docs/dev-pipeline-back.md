# Пайплайн разработки с нейросетями (notetaker-back)

Руководство для разработчика и AI-ассистента по совместной разработке функционала на стеке **Cloudflare Workers + Hono + D1 + Drizzle + Zod**.

---

## Выбор пайплайна

| Масштаб | Триггер | Пайплайн |
|---------|---------|----------|
| **Мелкий** | Фикс бага, 1–2 файла, < 50 строк | [Quick Pipeline](#quick-pipeline) |
| **Средний** | Новый роут/сервис, 3–5 файлов, без новых таблиц | [Quick Pipeline](#quick-pipeline) |
| **Крупный** | Новый домен, > 5 файлов, новые таблицы D1, новый AI-флоу | [Full Pipeline](#full-pipeline) |

# Quick Pipeline

Укороченный пайплайн для мелких/средних изменений.

## Когда использовать

- Фикс бага в существующем роуте/сервисе
- Добавление нового эндпоинта в существующий домен
- Новый запрос в `db/` без миграции
- Рефакторинг без изменения слоёв

## Шаги

### Q1. Краткий анализ

```
📍 QUICK PIPELINE: Анализ задачи

**Задача:** [краткое описание]
**Воркер(ы):** notetaker-<name> [+ notetaker-<name>, если задача затрагивает gateway+internal]
**Масштаб:** Мелкий / Средний
**Затрагиваемые файлы:** ~N файлов

Начинаю реализацию.
```

> Если задача требует нового воркера — это **не** Quick Pipeline. Переключайся на Full Pipeline и согласуй создание воркера через `pragmatic-architect` (чтобы убедиться, что нельзя встроить в существующий).

### Q2. Реализация + проверка агентами

Нейросеть:
1. Пишет код
2. После написания **автоматически** вызывает агентов:
   - 🤖 `api-guardian` — если затронуты роуты/сервисы/работа с D1
   - 🤖 `clean-code-guardian` — если изменён файл > 50 строк
3. Исправляет замечания

```
📍 QUICK PIPELINE: Реализация

[Код...]

🤖 Агенты:
- api-guardian: ✅ Пройдено / ⚠️ Исправлено
- clean-code-guardian: ✅ Пройдено / ⚠️ Исправлено

Запусти `npm run typecheck` и проверь локально через `wrangler dev`.
```

### Q3. Проверка и коммит

Разработчик:
1. `npm run typecheck` (или `tsc --noEmit`)
2. `npm run lint` (если настроен)
3. Локальный smoke-тест через `wrangler dev`
4. Коммит

```
📍 QUICK PIPELINE: Готово

Изменения готовы. Сделай коммит:
`git add . && git commit -m "тип(scope): описание"`
```

# Full Pipeline

Полный пайплайн для крупных фич (новый домен, новые таблицы D1, новый AI-флоу).

---

## Фаза 0: Подготовка

### Разработчик
1. Создать ветку: `git checkout -b feature/название-фичи`
2. Сформулировать цель в 2–3 предложениях
3. Определить Definition of Done
4. Определить, в **каком воркере (или каких воркерах)** живёт фича. Если кажется, что нужен новый воркер — обсудить с `pragmatic-architect` (новый воркер = только если домен реально новый и не вписывается в существующие).

### Нейросеть
1. Изучить контекст:
   - `notetaker-back/CLAUDE.md` — стек, архитектура микросервисов, слои внутри воркера, ключевые паттерны
   - Целевые воркеры (`notetaker-<name>/src/db/schema.ts`, `notetaker-<name>/src/routes/*`, `notetaker-<name>/src/services/*`) — текущая схема, существующие домены как образец
   - `wrangler.toml` каждого затронутого воркера — биндинги, Service Bindings, переменные
   - `api-gateway/src/routes/*` — как gateway маршрутизирует запросы
   - `docs/modules/<module>.md` — если домен уже существует
2. Не предлагать решений, противоречащих существующей архитектуре. Особенно: не дублировать JWT-проверку в internal-воркерах, не вводить `env.AI: Ai` нигде кроме `ai`, не звать чужие воркеры по HTTP.
3. Проверить, нужна ли миграция D1 (если новые таблицы/колонки — да, через `drizzle-kit generate` в воркере, владеющем таблицами).
4. Проверить, нужен ли новый Service Binding (gateway → internal). Если да — обновить `wrangler.toml` обоих воркеров и тип `Env` в gateway.
5. Проверить, нужны ли изменения в Vectorize:
   - Новая сущность, которую надо искать по смыслу → `upsert/delete` в `ai` после CRUD в исходном воркере (через SVC binding и `c.executionCtx.waitUntil(...)`).
   - Изменение embedding-модели или dimensions — это **новый индекс**, не миграция: `wrangler vectorize create <new-name> ...`, новый `index_name` в `wrangler.toml`, реиндекс. До появления реальной потребности — не трогаем.
   - В query всегда `namespace = userId` + `filter: { userId }` — никаких cross-user поисков.

### 2.2 Ревью плана разработчиком

**Критерии хорошего плана:**
- Без over-engineering
- Использует слои `routes → services → db`
- Сервисы остаются HTTP-agnostic
- Минимальные изменения схемы D1

### 2.3 Реализация

Нейросеть:
- Пошагово реализует план
- `npm run typecheck` после изменений типов
- Если есть миграция: `npx drizzle-kit generate` → проверить SQL → `wrangler d1 migrations apply <DB> --local`

### 2.4 Ручное тестирование

Чек-лист от нейросети:
```markdown
## Чек-лист этапа N

### Позитив
- [ ] Сценарий 1: запрос → ожидаемый ответ
- [ ] Сценарий 2: ...

### Негатив
- [ ] Невалидный JSON → 400 + сообщение Zod
- [ ] Без JWT → 401
- [ ] Чужой ресурс → 403/404

### Edge
- [ ] Пустые поля
- [ ] Превышение лимитов (длина текста, размер payload)
- [ ] Параллельные запросы (если важно)
- [ ] AI-флоу: таймаут `env.AI.run`, неуспешный ответ модели, лимиты Workers AI
```

Тестирование локально через `wrangler dev` + curl/Postman/REST-клиент.

### 2.5 Code Review

Нейросеть проверяет:

1. **Слои** — нет утечек (HTTP в сервисе, SQL в роуте)
2. **Дублирование** — общие хелперы в `lib/`
3. **Безопасность** — Zod на всех входах, JWT на защищённых роутах, секреты только в `wrangler.toml`/dashboard, не в коде
4. **Ошибки** — `Result<T>` или централизованный `app.onError`, нет голых `throw new Error('...')` для ожидаемых ситуаций
5. **TypeScript** — нет `any`, типы инферятся из Zod, `Env` типизирован
6. **Производительность** — фоновая работа через `c.executionCtx.waitUntil()`, нет блокирующих ожиданий там, где можно ответить сразу

### 2.6 Исправления

Нейросеть исправляет замечания, объясняет каждое.

## Фаза 3: Завершение фичи

### Интеграционная проверка
- Полный пользовательский сценарий через `wrangler dev` (gateway + internal-воркеры подняты параллельно — gateway проксирует Service Bindings).
- Проверка миграций на чистой локальной D1
- `wrangler deploy --dry-run` в каждом затронутом воркере — нет ошибок сборки.
- **Биндинги в `wrangler.toml` совпадают с типом `Env`** в каждом воркере:
  - `api-gateway`: `JWT_SECRET` + Service Bindings (`AUTH`, `NOTES`, `AI`, `PARSER`, `PROJECTS`).
  - `ai`: `DB` (D1), `AI` (Workers AI), `VECTORIZE` (`index_name = "notetaker-vectors"`). Никаких `JWT_SECRET`.
  - Internal-воркеры: только нужные таблицы D1, без публичного `[[routes]]`, без `[ai]`, без `[[vectorize]]`.
- **Service Bindings двусторонне согласованы** — `service` в gateway-`wrangler.toml` указывает на ровно тот `name`, что прописан в `wrangler.toml` целевого воркера.
- **AI-флоу прогнан локально через `wrangler dev`** в `ai` — Workers AI работает в dev через прокси к Cloudflare, нужен `wrangler login`.
- **Vectorize-индекс существует** (`wrangler vectorize list` показывает `notetaker-vectors` с `dimensions=1024, metric=cosine`) и `index_name` в `ai/wrangler.toml` совпадает.
- **Vector-флоу прогнан:** заметка создаётся → через `c.executionCtx.waitUntil(...)` появляется в Vectorize (`wrangler vectorize get-by-ids notetaker-vectors --ids note:<id>`); удаление заметки → запись пропадает; query возвращает только записи из правильного namespace.
- **Гибрид настроек проверен:** дефолтные промпт/модель отрабатывают без записи в D1; запись в D1 переопределяет дефолт; невалидная модель в D1 → fallback на `DEFAULT_MODEL`. Embedding-модель остаётся константой.
- Все секреты заведены через `wrangler secret put <KEY>` в нужных воркерах (не в `wrangler.toml`). `JWT_SECRET` — только в `api-gateway` и `auth` (последний для подписи).

## Антипаттерны

1. **Бизнес-логика в роуте** — всегда выноси в сервис
2. **`Context` / `Request` в сигнатуре сервиса** — сервис должен быть HTTP-agnostic
3. **Прямой `c.env.DB.prepare(...)`** в сервисе — только через `db/`
4. **HTTP-вызов между своими воркерами** — только Service Bindings
5. **`env.AI: Ai` в воркере, не являющемся `ai`** — все AI-вызовы маршрутизируются через ai-воркер
6. **`env.VECTORIZE` (или `[[vectorize]]` в `wrangler.toml`) в воркере, не являющемся `ai`** — все vector-операции через ai-воркер
7. **Vectorize query без `namespace=userId`** — риск утечки данных между пользователями
8. **Дублирование векторов в D1** — Vectorize и есть хранилище
9. **`BaseVectorRepository` / generic-обёртка над Vectorize** — нативный API из 4 методов короткий
10. **JWT-проверка во внутреннем воркере** — только в `api-gateway`
11. **Internal-воркер с публичным `[[routes]]` в `wrangler.toml`** — все запросы только через gateway
12. **Новый воркер под мелкую задачу** — встраивай в существующий
13. **Хардкод промптов** — гибрид `config/prompts.ts` + D1
14. **Игнорирование агентов** — замечания обязательны к разбору
15. **Большие этапы** — > 4 часов работы дроби на части
16. **Коммиты "fix", "update"** — пиши scope и суть
17. **Общий `try/catch` ради формата ответа** — есть `app.onError`
18. **Node-only зависимости** — Workers их не любят, проверь edge-совместимость
